# Claude Codeの基本構造

## はじめに

前回の記事では、Claude Codeの基本的な使い方と最低限覚えておくべきコマンドを学びました。今回は、Claude Codeをより深く使いこなすために、その「基本構造」を理解していきます。

Claude Codeは単なるチャットツールではなく、設定ファイルやディレクトリ構造によって動作をカスタマイズできる仕組みを持っています。この構造を理解することで、自分のプロジェクトに合わせた最適な環境を構築できるようになります。

なお、この記事では「ホームディレクトリ」「プロジェクトルート」といったディレクトリ構成の概念や、ターミナルでのコマンド操作が登場します。これらの知識がなくても記事を読み進めることはできますが、正確に理解するにはディレクトリ構成やCLIコマンドの基礎知識があると望ましいです。ニーズがあれば、今後これらの基礎知識をまとめた記事も準備する予定です。

## .claudeディレクトリの構造

Claude Codeの設定は、大きく分けて2つの場所に保存されます。「ホームディレクトリ」と「プロジェクトルート」です。それぞれの役割を理解しておきましょう。

### ホームディレクトリ（~/.claude/）

ホームディレクトリの .claude フォルダには、ユーザー全体に適用される設定が格納されます。ここに置いた設定は、どのプロジェクトでClaude Codeを起動しても有効になります。

主なファイルとフォルダは以下の通りです。

**CLAUDE.md**
全プロジェクト共通のグローバルな指示ファイルです。たとえば「コメントは常に日本語で書く」「コミットメッセージは日本語にする」のように、どのプロジェクトでも共通して守ってほしいルールを記述します。

**settings.json**
ツールの許可設定やAPI設定など、ユーザーレベルの設定ファイルです。

この他にも、認証情報やメモリ用のファイルが自動生成されますが、ユーザーが意識すべきは上記の2つです。

### プロジェクトルート（.claude/）

各プロジェクトのルートディレクトリにも .claude フォルダを置くことができます。ここにはプロジェクト固有の設定を格納します。

**CLAUDE.md**
プロジェクト固有の指示ファイルです。前回の記事でも紹介しましたが、これが最も重要な設定ファイルです。プロジェクトの技術スタック、コーディングルール、ディレクトリ構造などを記述します。

**.claude/settings.json**
プロジェクト固有の設定ファイルです。そのプロジェクトでのみ有効な許可設定などを記述します。

**.claude/commands/**
プロジェクト固有のカスタムコマンドを定義するフォルダです。チームで共有したいコマンドをここに置きます。

### 設定の優先順位

設定が重複した場合、以下の優先順位で適用されます。

**プロジェクト設定 > ユーザー設定 > デフォルト**

つまり、プロジェクトルートのCLAUDE.mdに書いた内容は、ホームディレクトリのCLAUDE.mdよりも優先されます。グローバルに「コメントは日本語で」と書いていても、特定のプロジェクトで「コメントは英語で」と書けば、そのプロジェクトでは英語のコメントが生成されます。

## settings.jsonの設定

settings.jsonは、Claude Codeの動作を細かく制御するための設定ファイルです。主要な設定項目を見ていきましょう。

### permissions（権限設定）

ツールの実行を許可するか拒否するかを事前に設定できます。毎回確認ダイアログが出るのが煩わしい場合に便利です。

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Bash(npm run *)"
    ],
    "deny": [
      "Bash(rm -rf *)"
    ]
  }
}
```

上記の例では、ファイルの読み取りと npm run で始まるコマンドは自動的に許可し、rm -rf のような危険なコマンドは自動的に拒否するよう設定しています。ワイルドカード（*）を使ったパターン指定にも対応しています。

### env（環境変数）

Claude Codeのセッション内で使用する環境変数を設定できます。

```json
{
  "env": {
    "NODE_ENV": "development"
  }
}
```

### mcpServers（MCP サーバー接続設定）

外部ツールと連携するためのMCPサーバーの接続設定です。MCPについては後の回で詳しく扱います。

## CLAUDE.mdの配置と優先順位

前回の記事でCLAUDE.mdの書き方を紹介しましたが、実はCLAUDE.mdは複数の場所に配置でき、それぞれ異なるスコープで適用されます。

### 3つの配置レベル

**グローバル（~/.claude/CLAUDE.md）**
全プロジェクトに共通して適用される指示です。自分の開発スタイルに関するルールを書いておくと便利です。

**プロジェクト（your-project/CLAUDE.md）**
特定のプロジェクトに適用される指示です。最も重要な配置場所であり、プロジェクトの技術スタックやルールを記述します。Gitで管理すれば、チームメンバー全員が同じルールを共有できます。

**サブディレクトリ（your-project/src/CLAUDE.md）**
特定のディレクトリ以下にのみ適用される指示です。たとえば、フロントエンドのディレクトリにだけ適用したいルールがある場合に使います。

### 使い分けの例

グローバルのCLAUDE.mdには、個人の好みや習慣を書きます。

```markdown
# グローバル設定

- コメントは日本語で記述する
- コミットメッセージは日本語にする
- テストを書いてから実装する（TDD）
```

プロジェクトのCLAUDE.mdには、そのプロジェクト固有の情報を書きます。

```markdown
# プロジェクト設定

## 技術スタック
- Next.js 14（App Router）
- TypeScript
- Tailwind CSS

## ルール
- コンポーネントは関数コンポーネントで作成する
- src/lib/ にユーティリティ関数を配置する
```

このように階層的に設定を管理することで、グローバルなルールを維持しつつ、プロジェクトごとに柔軟なカスタマイズが可能になります。

## 権限システム

Claude Codeは安全性に配慮して、操作の種類ごとに異なる権限レベルを設定しています。

### 読み取り（Read）

ファイルの内容を読む操作です。デフォルトで許可されており、確認なしで実行されます。AIがプロジェクトの構造やコードを理解するために必要な操作なので、自動的に許可されています。

### 書き込み（Write / Edit）

ファイルの作成や編集を行う操作です。実行前にdiff（変更差分）が表示され、ユーザーの確認が必要です。どの行がどう変わるのかを確認してから承認できるので、意図しない変更を防げます。

### コマンド実行（Bash）

ターミナルコマンドの実行です。実行するコマンドの内容が事前に表示され、ユーザーの確認が必要です。npm install のような安全なコマンドもあれば、rm -rf のような危険なコマンドもあるため、内容をよく確認してから承認しましょう。

### 事前設定で確認を省略する

よく使う安全な操作については、前述のsettings.jsonで事前に許可設定をしておくことで、毎回の確認を省略できます。ただし、危険な操作まで許可してしまわないよう注意が必要です。

## 非対話モード（-p / --print）

通常のClaude Codeは対話形式で使いますが、ワンショットで質問や指示を投げて結果だけ受け取る「非対話モード」も用意されています。

```bash
claude -p "package.jsonの依存関係を一覧にして"
```

このモードは以下のような場面で活躍します。

**スクリプトへの組み込み**
シェルスクリプトやCI/CDパイプラインの中にClaude Codeを組み込むことができます。たとえば、コミット前に自動でコードレビューを実行するといった使い方が可能です。

**パイプ処理**
他のコマンドの出力をClaude Codeに渡して処理させることもできます。

```bash
cat error.log | claude -p "このエラーログの原因を分析して"
```

対話モードのように会話を続ける必要がない、単発のタスクに向いています。

## 典型的な開発ワークフロー

ここまで学んだ構造を踏まえて、Claude Codeを使った典型的な開発フローを整理します。

### 新機能開発の流れ

**ステップ1：プロジェクトの理解**
まず、AIにプロジェクトの全体像を理解させます。CLAUDE.mdが用意されていれば自動で読み込まれますが、特定のファイルを参照させたい場合は@メンションを使います。

```
> @README.md このプロジェクトの構成を教えて
```

**ステップ2：計画の策定**
/plan コマンドやShift+Tabでプランモードに切り替え、実装方針をAIとすり合わせます。

```
> /plan ユーザー認証機能を追加したい
```

**ステップ3：実装の実行**
計画に基づいて実装を進めます。AIがファイルの作成・編集を提案するので、diff を確認しながら承認していきます。

**ステップ4：テストと検証**
実装が完了したら、テストの実行を指示します。

```
> テストを実行して、問題があれば修正して
```

**ステップ5：コミットとPR作成**
問題がなければ、Gitへのコミットやプルリクエストの作成もAIに任せられます。

```
> 今回の変更をコミットして
```

### 効率的に使うコツ

**具体的に指示する**
「いい感じにして」ではなく、「ヘッダーの背景色を青に変えて、フォントサイズを16pxにして」のように具体的に伝えましょう。

**CLAUDE.mdを育てる**
使いながら気づいたルールや注意点をCLAUDE.mdに追記していきましょう。CLAUDE.mdは一度書いて終わりではなく、プロジェクトと一緒に成長させるものです。

**段階的に進める**
大きな機能をいきなり作らせるのではなく、小さなステップに分けて進めましょう。一つずつ確認しながら進めることで、手戻りを最小限に抑えられます。

**diffを確認する習慣をつける**
AIが提案する変更内容は必ず確認しましょう。特に既存のコードを修正する場合は、意図しない変更が含まれていないか注意が必要です。

## まとめ

この記事では、Claude Codeの基本構造を学びました。

**ディレクトリ構造**
- ~/.claude/（ホーム）にはグローバル設定、プロジェクトルートにはプロジェクト固有の設定を配置する
- 設定の優先順位は「プロジェクト > ユーザー > デフォルト」

**設定ファイル**
- settings.json で権限やMCPサーバーを設定できる
- CLAUDE.md はグローバル、プロジェクト、サブディレクトリの3レベルで配置可能

**権限システム**
- 読み取りはデフォルト許可、書き込みとコマンド実行は確認が必要
- settings.json で事前に許可・拒否ルールを設定できる

**非対話モード**
- claude -p でワンショット実行が可能
- スクリプトやCI/CDへの組み込みに活用できる

Claude Codeの構造を理解することで、ただ使うだけでなく、自分の開発スタイルに合わせた環境を構築できるようになります。次回は、Claude Codeを使ってより実践的なアプリケーション開発に取り組んでいきます。
