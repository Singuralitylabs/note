# Claude CodeのHooksで自動化を仕込む

## はじめに

前回の記事では、Claude Codeに自分専用のワークフローを覚えさせる「Skills」を紹介しました。今回は、Claude Codeのエージェントループの特定タイミングで「必ず実行したい処理」を仕込める仕組み、**Hooks** を紹介します。

「ファイルを編集したら毎回フォーマッタを走らせたい」「長いタスクが終わったら音で知らせてほしい」「危険なコマンドはClaudeに実行させたくない」――こうした願いを、設定ファイル一枚で叶えられる機能です。

## Hooksとは何か

Hooksは、Claude Codeのエージェントループ内で特定のイベントが発生したときに、**シェルコマンドを決定論的に実行する仕組み** です。

ここで重要なのは「決定論的」という点です。SkillsやMCPがLLMの判断を経由するのに対し、Hooksは **LLMを介さずに、設定したとおりに必ず実行されます**。毎回確実に同じ処理が走るので、フォーマッタやlintのような定型処理に最適です。

LLMを呼び出さないため、コンテキストもほとんど消費しません。「忘れず確実に走らせたい処理」をClaudeの判断に委ねず、機械的に挟み込めるのがHooksの強みです。

公式ドキュメントは[Anthropicの公式サイト](https://code.claude.com/docs/en/hooks)で公開されています。

## 主要なイベントタイプ

Hooksはエージェントループの随所に発火ポイントが用意されています。よく使うものを4つだけ紹介します。

**SessionStart** はセッション開始時に発火します。環境チェックやGit状態の確認に向きます。

**UserPromptSubmit** はユーザーがプロンプトを送信した直後に発火します。入力バリデーションや機密語のフィルタに使えます。

**PreToolUse** はツール実行の直前に発火します。`exit 2` で処理を中断できるため、危険なコマンドのブロックに利用できます。

**PostToolUse** はツール実行の直後に発火します。ファイル編集後の自動フォーマットや自動lintの定番ポイントです。

このほかにも `Stop`（応答完了時）、`SessionEnd`（セッション終了時）、`PreCompact`（コンテキスト圧縮前）など多数のイベントがあります。

## 設定方法

Hooksは `settings.json` の `hooks` フィールドに書きます。チームで共有したい設定はプロジェクトの **.claude/settings.json**、自分だけ試したい設定は **.claude/settings.local.json**（gitignore対象）に置くのが基本です。

下記は「Claudeがファイルを編集（WriteまたはEdit）したら、毎回Prettierを自動実行する」という設定例です。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "FILE=$(jq -r '.tool_input.file_path') && npx prettier --write \"$FILE\""
          }
        ]
      }
    ]
  }
}
```

ポイントは2つあります。

**matcher** は対象ツール名のパターンです。`Write|Edit` のようにパイプで複数指定できます。省略すると全ツールにマッチします。

**command** は実行するシェルコマンドです。Hooks発火時、ツール入力やセッション情報は標準入力にJSONで渡されるので、`jq` で必要な値を抽出して使うのが定番パターンです。上の例では編集対象のファイルパスを取り出してPrettierに渡しています。

## 実用例

### ファイル編集後の自動フォーマット

先ほどのPrettier例がまさに定番です。Claudeがどのファイルを触っても、保存スタイルが必ず統一されます。フォーマット結果がClaudeに見えなくても、ファイルは確実に整形されている状態を保てます。

### 応答完了時の効果音通知

長めのタスクをClaudeに任せて他の作業をしているとき、終わったタイミングを音で気付けるようにする使い方です。`Stop` イベントを使います。

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "afplay /System/Library/Sounds/Glass.aiff"
          }
        ]
      }
    ]
  }
}
```

`afplay` はmacOS組み込みのコマンドです。Linuxなら `paplay`、Windowsなら別の方法に置き換えてください。

### 危険なコマンドのブロック

`PreToolUse` で `exit 2` を返すと、ツール実行そのものを中止できます。たとえば `rm -rf` を含むBashコマンドを問答無用で止める設定はこう書けます。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command' | grep -q 'rm -rf' && exit 2 || exit 0"
          }
        ]
      }
    ]
  }
}
```

Claudeがどう判断したかに関わらず、機械的に弾けるのが安心ポイントです。

## 動作確認のコツ

Claude Code内で **/hooks** と打つと、現在ロードされている全Hooksが一覧表示されます。設定したのに動かないときは、まずここで読み込まれているか確認してください。表示されない場合はJSONの構文エラーやパスの誤りを疑います。

なお `/hooks` は閲覧専用で、追加や編集はできません。

トラブル切り分けのために一時的にすべてのHooksを止めたい場合は、`settings.json` に下記を追加します。

```json
{
  "disableAllHooks": true
}
```

確認が終わったら忘れずに削除しましょう。

## SkillsとHooksの使い分け

最後に、Skills（前回紹介）とHooksの位置づけを整理しておきます。

**Skills** は「必要なときだけClaudeが判断して呼び出す」ワークフローです。柔軟ですが、毎回呼ばれるとは限りません。

**Hooks** は「特定イベントで必ず走る」処理です。確実性は高いですが、Claudeの判断は介在しません。

「コードレビューを依頼されたらこの観点で見てほしい」はSkills、「ファイルを編集したら必ずフォーマットしたい」はHooks、と覚えておくと迷いません。両者は競合するものではなく、組み合わせて使うものです。

## まとめ

Hooksは、Claude Codeに「忘れず確実にやってほしい処理」を仕込むための仕組みです。

設定ファイル一枚で、フォーマット、通知、ブロックといった定型処理を自動化できます。LLMを介さないのでコストもかからず、結果も予測可能です。

まずは自動フォーマットや応答完了の通知音といった、効果が分かりやすいものから試してみるのがおすすめです。
